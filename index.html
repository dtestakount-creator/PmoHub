            About the Tool

            Operating guide, logic reference, and data model

            Purpose and intended use

            This workspace is a programme-specific control hub for the DKTAI programme. It is designed to give the PMO and workstream leads one place to track delivery, governance, actions, RAID, decisions, milestones, stakeholders, and reporting without the overhead of a heavy enterprise platform.

            It is optimized for practical programme control: identify where pressure is building, understand why a workstream is moving, and extract a clear status view quickly.

            Best fit: internal programme governance, workstream reviews, PMO coordination, SteerCo pack preparation, and maintaining a single local working dataset.

            Metric logic and scoring model

            Dashboard KPIs

            • Total workstreams: count of all project/workstream records.
            • On Track / At Risk / Not Started: direct counts based on each workstream status. The Not Started tile also groups delayed workstreams into the urgent-attention bucket.
            • High RAID: open RAID items with priority = High and status not done.
            • Overdue actions: actions with a due date earlier than today and status not done.
            • Programme completion: average of three completion rates: completed milestones %, completed actions %, and resolved RAID %.

            Programme health hero

            • Overall RAG: driven first by workstream status. Any At Risk workstream makes the top-level RAG red. If there are no at-risk workstreams but delays/overdues accumulate, the overall status turns delayed/amber.
            • Base health score: weighted from project delivery 35%, milestone completion 20%, action completion 20%, RAID resolution 15%, and average workstream progress 10%.
            • Deductions: penalties are applied for at-risk workstreams, delayed workstreams, high-priority open RAID, overdue milestones, and action overdues.
            • Result: the final score is capped between 5 and 100 so the health gauge remains usable even in stressed states.
            This means the tool does not treat progress % alone as truth. It blends delivery progress with actual governance stress signals such as overdue actions, unresolved RAID, and milestone slippage.

            Decisions logic

            The Decision Log is centralized, but decisions can be scoped either to the Programme or to a specific Project / Workstream.

            • A workstream decision appears both in the central repository and in the local Decisions tab of the linked workstream.
            • A programme decision stays global and is not forced into any workstream unless explicitly linked later.
            • Decision records are meant to explain why scope, timing, cost, or risk moved — not just that they moved.

            Reports and outputs

            Extract Report generates a leadership-facing summary based on the current dataset. It is intended for a fast programme snapshot, not a full archival export.

            Dashboard cards are designed to support verbal status reporting: what is under pressure, what is overdue, what decisions matter, and what governance comes next.

            Delivery Bank acts as a proof/evidence layer for completed items, which helps when moving from “status claim” to “demonstrable output”.

            Data model

            EntityKey fields
            Projectsid, layer, name, owner, PMO lead, status, progress, dates, objective, scope, deliverables, stakeholders
            RAIDid, type, title, priority, owner, projectId, due, status, description, mitigation, dependency-link fields
            Actionsid, task, owner, projectId, due, priority, status, notes
            Decisionsid, title, description, scope, projectId, status, owner, date, impact, change drivers
            Milestonesid, name, date, status, projectId, owner, notes, proof link
            Meetingsid, title, date, time, duration, type, recurrence, location, description
            Stakeholdersprogramme people, workstream stakeholders, regional and SSL stakeholder records
            Persistence model: browser localStorage. This is a local working file model, not a hosted multi-user database.

            JSON import/export

            • Export JSON creates a full backup of the working dataset, including projects, RAID, actions, milestones, meetings, decisions, and stakeholder structures.
            • Import JSON restores a saved dataset into the current workspace.
            • Import expects a valid DKTAI PMO export structure. Unknown or missing core structures are rejected.
            • Exports also include metadata such as export timestamp/version to support restore and auditability.

            Excel upload and sync

            Excel button is for bulk upload of one table type at a time: Actions, RAID, or Milestones. The file type is auto-detected from headers.

            • Supported upload files: .xlsx, .xls, .csv
            • The first row must contain headers.
            • Project matching uses project_name against existing workstream names. If no exact match exists, the item is loaded at programme level.

            Sync button opens the full data architecture modal. From there you can:

            • Export the full Excel data model as a multi-sheet workbook.
            • Import a filled workbook back to update the programme.
            • Export or import the full JSON backup.
            • See the local storage key and the last Excel sync timestamp.

            Operational notes

            • This tool is programme-specific and intentionally opinionated around DKTAI operating needs.
            • It is strongest as a PMO operating hub, working file, and reporting layer.
            • It is not a hosted system of record with permissions, workflow approvals, or multi-user audit trails.
            • The most important control principle in the tool is linkage: workstreams → RAID/actions/milestones/decisions → dashboard/reporting.
